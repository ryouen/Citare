# Citare Claim Extraction Prompt v0.1 (baseline)

Temperature = 0. Effort = default.

## Purpose of This Task

You will read an academic paper and extract its "claims" as structured JSON for the Citare Graph database.

**Why this matters:**
The same paper is read millions of times by researchers and AI agents worldwide. Once structured, it never needs to be re-processed. The more accurate and comprehensive your extraction, the more every subsequent user benefits.

**The two errors we most want to prevent:**
1. **Causal misattribution** — writing "causes" when the paper only shows "is associated with." The causal_strength field prevents this.
2. **Missing important claims** — a key finding absent from the structured data. Missing claims are extremely hard to detect after the fact. When in doubt, output with a lower confidence_score rather than omitting.

---

## PDF Processing Note

The PDF may have been pre-processed (images removed to preserve text layer). Read all text content, tables, and equations from the text layer. If figures are referenced but not visible, note this in the relevant claim's source_text.

## Output Structure

Output everything as a single valid JSON object.

### 1. paper (metadata + study-level defaults)

```json
{
  "paper": {
    "doi": "10.xxxx/yyyy",
    "title": "Paper title",
    "title_concepts": ["concept_a", "concept_b"],
    "authors": ["Author A", "Author B"],
    "year": 2024,
    "venue": "Journal name",
    "paper_type": "empirical / conceptual / review / meta_analysis / book / book_chapter",
    "domain": "e.g. organizational_behavior, machine_learning, ai_safety",

    "default_method": {
      "sample_size": 51,
      "unit_of_analysis": "e.g. team, individual, organization, computational, gene, national, dyadic",
      "industry": "e.g. manufacturing, healthcare, mixed",
      "country": "e.g. US, JP, multinational",
      "study_design": "e.g. cross-sectional survey, RCT, benchmark evaluation"
    },

    "default_causal_strength": {
      "design_basis": "cross_sectional / longitudinal / quasi_experimental / rct / meta_analysis / computational_demonstration / theoretical",
      "temporal_precedence": "none / partial / full",
      "author_framing": "causal / associational / suggestive / existence_proof",
      "manipulation_of_iv": false
    }
  }
}
```

**default_method** and **default_causal_strength** describe the primary study design. Most claims in a single paper share the same research design and sample. Individual claims only need to override these when they differ (e.g., a different unit_of_analysis for one analysis, or a Discussion-only claim with design_basis = "theoretical").

**title_concepts:** List every concept named in the title or centrally defined in the abstract, using `{unit_of_analysis}_{concept}` format. Each must have a corresponding DEFINITION claim in the claims array.

DOI: Search the PDF's headers, footers, and body text. If not printed but confidently inferrable, record with `"doi_source": "inferred"`. Use null only when truly unidentifiable.

source_page (all claims): Use the **printed page number** from headers/footers, not the PDF viewer's page count. If absent, use PDF page numbers with `"source_page_note": "PDF page number"`.

### 2. claims — four templates

The atomic unit of Citare. **Consider at least one claim per section and per Table/Figure.**

Minimum counts: empirical ≥ 5, conceptual ≥ 8.

#### Common fields

```json
{
  "id": "{first_author_surname_lowercase}{year}_{def/rel/exist/meta}{number}",
  "template_type": "DEFINITION / RELATION / EXISTENCE_CLAIM / META_CLAIM",
  "l0_json": {},
  "l3_json": {},
  "source_text": "Direct quote of 1-3 sentences from the paper. No paraphrasing.",
  "source_page": 354,
  "source_section": "Section name",
  "evidence_type": "See Appendix A",
  "verification_status": "verified_in_paper / proposed_in_paper / null",
  "confidence_score": 0.90
}
```

**No l2 or l1 fields.** Citare clients generate natural-language summaries and triples from l0_json + source_text at read time. This avoids redundant output.

#### DEFINITION — "What X is"

```json
"l0_json": {
  "concept": "{unit_of_analysis}_{concept}",
  "key_elements": ["element1", "element2", "element3"],
  "distinguished_from": ["concept it could be confused with"]
}
```

verification_status = null.

For framework papers: create **separate DEFINITIONs for each component**, not just one for the whole framework.

#### RELATION — "X is related to Y"

```json
"l0_json": {
  "iv": "{unit_of_analysis}_{concept}",
  "dv": "{unit_of_analysis}_{concept}",
  "relation": "positive / negative / null / curvilinear / conditional",
  "mediator": null,
  "moderator": null,
  "author_generalization": null
}
```

**author_generalization:** null when Discussion and Method use the same terms. When the author uses a more abstract label in Discussion than what was measured in Method:

```json
"author_generalization": {
  "discussion_label": "wellbeing",
  "actual_measure": "individual_life_satisfaction_diener_swls"
}
```

**causal_strength override:** Include this field in a claim **only** when it differs from paper.default_causal_strength. For example, a Discussion-only claim might override design_basis to "theoretical" while all Results claims use the paper default. Only include the keys that differ. Omit the field entirely when the paper default applies.

```json
"causal_strength": {
  "design_basis": "theoretical",
  "author_framing": "suggestive"
}
```

**method_metadata override:** Include **only** when different from paper.default_method. For example, one analysis uses individual-level data while the paper default is team-level. Only include the keys that differ. Omit the field entirely when the paper default applies.

```json
"method_metadata": {
  "sample_size": 427,
  "unit_of_analysis": "individual"
}
```

**l3_json:** Record only applicable fields (full list in Appendix B). Always reference the paper's Tables directly and record **all comparison baselines**. Record equations in `formal`. For conceptual papers, record structural information in `additional`.

#### EXISTENCE_CLAIM — "X exists / was observed / has a limitation"

EXISTENCE_CLAIMs capture findings that are not variable relationships but are critical for correct interpretation — without them, downstream users may over-interpret RELATION claims. Study design limitations, boundary conditions, and methodological validations belong here.

```json
"l0_json": {
  "phenomenon": "phenomenon name",
  "evidence": "1-2 sentence summary of evidence"
}
```

**Commonly overlooked but important cases:**
- Study design limitations ("The cross-sectional design prevented a demonstration of causality")
- Paper motivation ("Current Web lacks machine-processable semantics")
- Capability demonstration ("Non-parametric memory can be hot-swapped")
- Boundary condition ("The method failed on personality tests")
- Non-relational numerical findings ("Significant between-team variance, ICC=.39")

#### META_CLAIM — "Integrative finding across studies or across the entire paper"

```json
"l0_json": {
  "integrated_finding": "Summary of integrated finding",
  "synthesis_type": "quantitative_meta_analysis / qualitative_synthesis / theoretical_integration",
  "k": null,
  "N_total": null
}
```

### 3. theories

Record only those **explicitly named** in the paper. Criteria: (1) organizes 3+ claims, (2) includes mechanism, (3) can generate hypotheses.

```json
"theories": [{
  "name": "Formal name",
  "type": "theory / theoretical_model / framework",
  "component_claims": ["claim ID list"],
  "description": "1-2 sentences",
  "scope": "Applicable range"
}]
```

### 4. measurement_methods

```json
"measurement_methods": [{
  "id": "short_name",
  "name": "Full name",
  "subtype": "e.g. questionnaire_scale, behavioral_observation, computational_benchmark, physiological_measure, experimental_paradigm, interview, content_analysis, archival_data",
  "measures": "concept name",
  "details": {
    "item_count": null,
    "response_scale": null,
    "reverse_items": [1, 3, 5],
    "reliability_reported": 0.82,
    "original_source": "Author, year",
    "copyright_status": "public / proprietary"
  },
  "used_in_claims": ["claim ID list"]
}]
```

`reverse_items`: list of 1-based item numbers, not item text (e.g., `[1, 3, 5]` not `["If you make a mistake..."]`).
`reliability_reported`: numeric value (e.g., `0.82`), not a string (not `"alpha = .82"`).

### 5. claim_relations

claim_relations are what make Citare a graph, not just a list of facts. Without them, users cannot trace how one finding builds on, qualifies, or contradicts another — the reasoning structure of the paper is lost. Record all relationships you find. Use confidence_score to express uncertainty rather than omitting.

```json
"claim_relations": [{
  "source_id": "claim ID",
  "target_id": "claim ID",
  "relation_type": "supports / extends / qualifies / contradicts / replicates / aggregates / part_of_model / background / apparent_tension",
  "incompleteness_category": "effect_disappears_under_control / hub_component / boundary_condition / extends_prior_definition / none",
  "confidence_score": 0.85
}]
```

**No context field.** The relationship meaning is fully expressed by relation_type + incompleteness_category + the content of the source and target claims.

Framework papers: connect components → whole with `part_of_model` + `hub_component`.
Motivation → proposal: connect with `background`.

### 6. paper_references

Record references that are substantively used in the paper, not bulk citation lists.

```json
"paper_references": [
  {"cited_doi": "10.xxxx/yyyy", "cited_title": "Title"},
  {"cited_doi": null, "cited_title": "Title only"}
]
```

**Record** if findings, definitions, or frameworks from the reference are used to build a claim; if results are compared or extended; or if cited with specific findings.

**Do not record** parenthetical lists without individual discussion. Typical count: 10-25.

DOIs: record only when certain. Do not fabricate.

## How to Read the Paper

Read in this order. Each step produces specific output.

**Step 1 — Title + Abstract.** Extract paper metadata including title_concepts. Paper motivations become EXISTENCE_CLAIMs.

**Step 2 — Method.** Extract default_method and default_causal_strength for the paper. Judge causal_strength by research design, not the author's word choice. Identify measurement_methods.

**Step 3 — Results + Tables + Figures.** Each Table maps to at least one RELATION or EXISTENCE_CLAIM. Read numbers directly from Tables. For each framework, create separate DEFINITIONs for each component, not one for the whole.

**Step 4 — Discussion.** Author interpretation, not data. Usually verification_status = "proposed_in_paper". Override causal_strength or method_metadata only when different from paper defaults.

**Step 5 — References.** Select 10-25 substantively used references. Record cited_doi only when certain.

## Judgment Criteria

**Template selection:**
```
"What X is"                                          → DEFINITION
"X is related to Y" (with or without statistics)     → RELATION
"X exists / was observed / has a limitation"          → EXISTENCE_CLAIM
"Integrative finding across studies or the paper"     → META_CLAIM
Citing another field's law as evidence for argument   → Independent RELATION (evidence_type: re_interpretation)
```

**causal_strength judgment (applied at paper.default_causal_strength level):**
- Judge by **research design**, not the author's word choice
- Benchmark comparisons / ablations → temporal_precedence = "none", author_framing = "existence_proof"
- "outperforms", "achieves SOTA" → author_framing = "existence_proof"
- "will be realized", "may influence" → author_framing = "suggestive"

**verification_status:**
- Backed by Results/data → `verified_in_paper`
- Author's interpretation / untested explanation → `proposed_in_paper`
- Fact and interpretation mixed → split into two claims
- DEFINITION → `null`

**iv/dv naming:**
- Use `{unit_of_analysis}_{concept}` format, where the prefix matches the study's unit of analysis (e.g. `team_`, `individual_`, `gene_`, `national_`)
- Use the paper's own terminology faithfully for the concept part
- If Appendix D lists a matching concept, prefer that name for consistency
- Based on **Method section measurements**, not Discussion abstractions
- Gap between Method and Discussion → record in author_generalization
- Measurement differences (self-report vs observer-rated) → measurement_methods, not iv/dv

## Appendix A: evidence_type (12 types)

```
meta_analysis, rct, quasi_experimental, longitudinal_field,
cross_sectional_field, qualitative_field, computational_demonstration,
theoretical_derivation, theoretical_integration, re_interpretation,
expert_framework, conceptual_argument
```

## Appendix B: l3_json Available Fields

Record only applicable fields. Do not fill with nulls.

**Statistical values:**
```
effect_size, effect_size_type, p, ci_lower, ci_upper, n,
r_squared, r_squared_type, k, N_total, heterogeneity_I2, publication_bias
```

**Structured data:**
```
mediation: {total_effect, direct_effect, indirect_effect, direct_p, indirect_p, proportion_mediated, note}
interaction: {variable: value}
models: [{label, effect_size, effect_size_type, ...}]  ← record ALL baselines
reliability: {alpha, icc, ...}
formal: {equation, variables, conditions, proof_type}
additional: {any structural information}  ← required for conceptual papers
```

**effect_size_type standard values:**

Psychology: `B, beta, r, rho, d, eta_squared, partial_eta_squared, omega_squared, odds_ratio, hazard_ratio`
ML: `BLEU, accuracy, F1, perplexity, Exact_Match, ROUGE, AUC, MAE, MSE`

Field usage: effect_size = one primary effect measure per claim. r_squared = overall model fit. reliability = scale reliability. additional = anything else.

## Appendix C: incompleteness_category (5 types)

```
effect_disappears_under_control — effect vanishes when controlling
hub_component                   — component of mediation/moderation model
boundary_condition              — boundary condition exists
extends_prior_definition        — extends a prior definition
none                            — can be cited standalone
```

## Appendix D: Concept Naming Reference

When a matching concept exists below, prefer that name for cross-paper consistency. Otherwise, use the paper's own terminology in `{unit_of_analysis}_{concept}` format.

### Organizational Behavior (team level)
```
team_psychological_safety, team_learning_behavior, team_performance,
team_context_support, team_leader_coaching, team_efficacy, team_type
```

### Prefix convention

Use the study's `unit_of_analysis` as the concept prefix. Examples: `team_`, `individual_`, `organizational_`, `computational_`, `gene_`, `national_`, `dyadic_`, `channel_`, `source_`.
