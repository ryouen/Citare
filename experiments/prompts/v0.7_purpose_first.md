# Citare Claim Extraction Prompt v0.7 (purpose-first)

Temperature = 0.

## Why this matters

The same paper is read by thousands of researchers and AI agents worldwide. Once structured correctly, it never needs re-processing. **Your accuracy becomes everyone's foundation**. The two errors to prevent above all:

1. **Causal misattribution** — writing "causes" when the paper only shows "is associated with." The `causal_strength` metadata exists specifically to prevent this.
2. **Missing key findings** — a crucial claim absent from the structured data cannot be recovered after the fact. **When in doubt, output with a lower `confidence_score` rather than omit**.

---

## How to read this PDF

Read in this order. Each step produces specific output:

1. **Title + Abstract** — paper metadata. **List `title_concepts` before writing any claims.** Every concept in the title or centrally defined in the abstract must correspond to a DEFINITION in `claims`.
2. **Method** — `default_method` and `default_causal_strength` for the paper. **Judge causal_strength by research design, not the author's word choice.** A cross-sectional study with causal framing is `design_basis=cross_sectional, author_framing=causal` — record both truthfully.
3. **Results + Tables + Figures** — each Table maps to at least one RELATION or EXISTENCE_CLAIM. Read numbers directly from Tables. Record **all baselines and comparison models**, not just the winner.
4. **Discussion** — author interpretation, not data. Usually `verification_status = proposed_in_paper`. Override `causal_strength` only when different from paper defaults.
5. **References** — 10-25 substantively used. Do not fabricate DOIs.

## Output: single JSON object

```json
{"paper": {...}, "claims": [...], "theories": [...], "measurement_methods": [...], "claim_relations": [...], "paper_references": [...]}
```

### 1. paper

```json
{
  "doi": "10.xxxx/yyyy",
  "title": "...",
  "title_concepts": ["{unit}_{concept}", "..."],
  "authors": ["..."],
  "year": 2024,
  "venue": "...",
  "paper_type": "empirical | conceptual | review | meta_analysis | book | book_chapter",
  "domain": "e.g. organizational_behavior, ml, ai_safety",
  "default_method": {
    "sample_size": N,
    "unit_of_analysis": "team | individual | organization | computational | gene | national | dyadic | ...",
    "industry": "...",
    "country": "...",
    "study_design": "..."
  },
  "default_causal_strength": {
    "design_basis": "cross_sectional | longitudinal | quasi_experimental | rct | meta_analysis | computational_demonstration | theoretical",
    "temporal_precedence": "none | partial | full",
    "author_framing": "causal | associational | suggestive | existence_proof",
    "manipulation_of_iv": true/false
  }
}
```

**`title_concepts` is written before claims** so that when you produce the claims array, each concept in `title_concepts` has an accountable DEFINITION. This is structural quality assurance — the output order enforces coverage.

DOI: look in headers, footers, body. If not printed but confidently inferrable, record with `"doi_source": "inferred"`. Use `null` only when **truly unidentifiable**, not when "uncertain."

### 2. claims — 4 templates

Every claim:

```json
{
  "id": "{first_author_surname_lc}{year}_{def|rel|exist|meta}{n}",
  "template_type": "DEFINITION | RELATION | EXISTENCE_CLAIM | META_CLAIM",
  "l0_json": {...},
  "l3_json": {...},
  "source_text": "1-3 sentence direct quote (no paraphrasing)",
  "source_page": <printed page number>,
  "source_section": "...",
  "evidence_type": "see Appendix A",
  "verification_status": "verified_in_paper | proposed_in_paper | null",
  "confidence_score": 0.0-1.0,
  "causal_strength": {...},
  "method_metadata": {...}
}
```

**No `l1` or `l2` fields.** Clients generate natural language and triples from `l0_json + source_text` at read time.

**Override rules**: `causal_strength` and `method_metadata` are included in a claim **only when they differ from paper defaults**. Omit entirely when paper default applies.

Minimum counts: empirical ≥ 5, conceptual ≥ 8. If a paper seems to warrant fewer, you are probably missing something — re-check Introduction (motivation → EXISTENCE_CLAIMs) and Discussion (proposed implications).

#### DEFINITION — "what X is"

```json
"l0_json": {
  "concept": "{unit}_{concept}",
  "key_elements": ["..."],
  "distinguished_from": ["..."]
}
```

`verification_status = null`. For framework papers proposing multiple components: **create a separate DEFINITION for each component**, not one DEFINITION describing the whole framework. This preserves queryability — later users will need "what is component X?" as a direct lookup.

#### RELATION — "X is related to Y"

```json
"l0_json": {
  "iv": "{unit}_{concept}",
  "dv": "{unit}_{concept}",
  "relation": "positive | negative | null | curvilinear | conditional",
  "mediator": null,
  "moderator": null,
  "author_generalization": null
}
```

`iv` and `dv` names are based on **what was actually measured in the Method section**, not the abstract label the author uses in Discussion. When Discussion uses a broader label than what was measured:

```json
"author_generalization": {
  "discussion_label": "wellbeing",
  "actual_measure": "individual_life_satisfaction_swls"
}
```

This matters because citation errors compound downstream. If the Method measured SWLS (life satisfaction), but the Discussion says "wellbeing", and we record iv=`individual_wellbeing`, then a future citation will claim "X affects wellbeing" when the paper actually showed "X affects a single life-satisfaction scale." The `author_generalization` field is our structural safeguard against this drift.

#### EXISTENCE_CLAIM — "X exists / was observed / has a limitation"

```json
"l0_json": {
  "phenomenon": "...",
  "evidence": "1-2 sentence evidence"
}
```

**EXISTENCE_CLAIMs are the most underextracted template.** They matter because they preserve:

- Study design limitations (cross-sectional admissions, sample-size caveats) → without these, downstream users over-interpret the RELATION claims
- Paper motivation / problem statement → the "why this paper exists" that frames its contributions
- Capability demonstrations ("Method achieves SOTA on benchmark X") → non-relational proofs
- Boundary conditions ("The effect did not hold for condition Y")
- Non-relational numerical findings (ICC, reliability, between-group variance)

When you encounter any of the above patterns, produce an EXISTENCE_CLAIM rather than trying to force it into RELATION.

#### META_CLAIM — "integrative finding across studies or across the paper"

```json
"l0_json": {
  "integrated_finding": "...",
  "synthesis_type": "quantitative_meta_analysis | qualitative_synthesis | theoretical_integration",
  "k": null,
  "N_total": null
}
```

### 3. theories

Record only **explicitly named** theories (organizes 3+ claims, includes mechanism, generates hypotheses):

```json
[{
  "name": "...",
  "type": "theory | theoretical_model | framework",
  "component_claims": ["claim_id_1", "claim_id_2"],
  "description": "1-2 sentences",
  "scope": "..."
}]
```

### 4. measurement_methods

```json
[{
  "id": "short_name",
  "name": "...",
  "subtype": "questionnaire_scale | behavioral_observation | computational_benchmark | physiological_measure | experimental_paradigm | interview | content_analysis | archival_data",
  "measures": "concept_name",
  "details": {
    "item_count": N,
    "response_scale": "...",
    "reverse_items": [1, 3, 5],
    "reliability_reported": 0.82,
    "original_source": "...",
    "copyright_status": "public | proprietary"
  },
  "used_in_claims": ["..."]
}]
```

`reverse_items`: 1-based item numbers, not item text. `reliability_reported`: numeric value (e.g., `0.82`), not a string.

### 5. claim_relations

**claim_relations are what make this a knowledge graph, not just a flat list of facts.** Without them, downstream users cannot trace how one finding builds on, qualifies, or contradicts another — the reasoning structure of the paper is lost. Two claims that are semantically related but not linked here will never be discovered together in search.

Record every relationship you find. Use `confidence_score` for uncertainty rather than omitting.

```json
[{
  "source_id": "...",
  "target_id": "...",
  "relation_type": "supports | extends | qualifies | contradicts | replicates | aggregates | part_of_model | background | apparent_tension",
  "incompleteness_category": "effect_disappears_under_control | hub_component | boundary_condition | extends_prior_definition | none",
  "confidence_score": 0.85
}]
```

**Framework papers**: connect each component → whole with `part_of_model` + `hub_component`.

**Motivation → proposal**: connect with `background`.

**Mediation findings** where a direct effect vanishes when controlling for a mediator: connect the bivariate claim → the mediated claim with `qualifies` + `incompleteness_category: effect_disappears_under_control`. This is the flag that prevents downstream misuse of standalone bivariate effects.

### 6. paper_references

Record references **substantively used** in the paper — findings, definitions, or frameworks built upon; results compared or extended; cited with specific findings. **Skip parenthetical bulk lists** (citations given only as en masse support without individual discussion).

```json
[{"cited_doi": "10.xxxx/yyyy", "cited_title": "..."}]
```

Typical count: 10-25. Do not fabricate DOIs. Use `null` if uncertain.

## Judgment Criteria

### Template selection

| pattern | template |
|---------|----------|
| "What X is" | DEFINITION |
| "X is related to Y" (with or without statistics) | RELATION |
| "X exists / was observed / has a limitation / achieves Y" | EXISTENCE_CLAIM |
| "Integrative finding across studies or the paper" | META_CLAIM |
| Citing another field's law as evidence for argument | Independent RELATION, `evidence_type: re_interpretation` |

### causal_strength (paper-level default)

Judge by **research design**, not word choice:

- Benchmark comparison / ablation studies → `temporal_precedence=none`, `author_framing=existence_proof`
- "outperforms", "achieves SOTA" → `author_framing=existence_proof`
- "will be realized", "may influence", "could enable" → `author_framing=suggestive`
- Cross-sectional data but author writes "causes" → record `design_basis=cross_sectional, author_framing=causal` (capture the gap)

### verification_status

- Backed by Results/data → `verified_in_paper`
- Author's interpretation or theoretical extension → `proposed_in_paper`
- Fact and interpretation intertwined → split into two claims
- DEFINITION → `null`

### iv/dv naming

- Use `{unit_of_analysis}_{concept}` format
- Unit prefix matches the study's unit of analysis (`team_`, `individual_`, `gene_`, `national_`, etc.)
- Use the paper's own terminology faithfully for the concept part
- Based on **Method section measurements**, not Discussion abstractions
- Measurement-method differences (self-report vs observer-rated) → `measurement_methods`, NOT `iv/dv`

## Appendix A: `evidence_type`

```
meta_analysis, rct, quasi_experimental, longitudinal_field,
cross_sectional_field, qualitative_field, computational_demonstration,
theoretical_derivation, theoretical_integration, re_interpretation,
expert_framework, conceptual_argument
```

## Appendix B: `l3_json` fields

Record only applicable fields. Do not fill with `null`s.

**Statistical values**: `effect_size`, `effect_size_type`, `p`, `ci_lower`, `ci_upper`, `n`, `r_squared`, `r_squared_type`, `k`, `N_total`, `heterogeneity_I2`, `publication_bias`

**Structured data**:
- `mediation`: `{total_effect, direct_effect, indirect_effect, direct_p, indirect_p, proportion_mediated, note}`
- `interaction`: `{variable: value}`
- `models`: `[{label, effect_size, effect_size_type, ...}]` ← **record ALL baselines, not just the winning one**
- `reliability`: `{alpha, icc, ...}`
- `formal`: `{equation, variables, conditions, proof_type}`
- `additional`: `{any structural information}` ← required for conceptual papers (no numerical effects to record otherwise)

**`effect_size_type` standard values**:

- Psychology: `B, beta, r, rho, d, eta_squared, partial_eta_squared, omega_squared, odds_ratio, hazard_ratio`
- ML: `BLEU, accuracy, F1, perplexity, Exact_Match, ROUGE, AUC, MAE, MSE`

## Appendix C: `incompleteness_category`

```
effect_disappears_under_control — effect vanishes when controlling for a mediator/confounder
hub_component                   — component of a mediation/moderation model; cite alone risks missing the hub
boundary_condition              — a boundary condition has been stated; generality is limited
extends_prior_definition        — extends a prior concept; original must be noted
none                            — can be cited standalone without missing context
```

## Appendix D: concept naming reference

`{unit_of_analysis}_{concept}`. Examples: `team_`, `individual_`, `organizational_`, `computational_`, `architecture_`, `channel_`, `source_`, `cognitive_`, `agent_`, `safety_`, `therapeutic_`, `gene_`, `national_`, `dyadic_`.

**Consistent naming hints (use when applicable)**:

- OB team-level: `team_psychological_safety`, `team_learning_behavior`, `team_performance`, `team_context_support`, `team_leader_coaching`, `team_efficacy`, `team_type`

If a concept matches one of the above, prefer that name for cross-paper consistency. Otherwise, use the paper's own terminology in the `{unit}_{concept}` format.
