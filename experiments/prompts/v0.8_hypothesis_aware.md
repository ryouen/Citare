# Citare Claim Extraction Prompt v0.8 (hypothesis-aware)

Temperature = 0.

## Why this matters

The same paper is read by thousands of researchers and AI agents worldwide. Once structured correctly, it never needs re-processing. **Your accuracy becomes everyone's foundation**. The errors that matter most:

1. **Causal misattribution** — writing "causes" when the paper only shows "is associated with." The `causal_strength` metadata exists specifically to prevent this.
2. **Missing key findings** — a crucial claim absent from the structured data cannot be recovered later. **When in doubt, output with a lower `confidence_score` rather than omit**.
3. **Dropping negative findings** — *a hypothesis that was explicitly tested and NOT supported is as important as one that was supported*. "X does not predict Y" is a claim. Papers that test 8 hypotheses must yield at least 8 hypothesis-RELATIONs, with `verification_status` reflecting the actual outcome for each.

## How to read this PDF

Read in this order. Each step produces specific output:

1. **Title + Abstract** — paper metadata. **List `title_concepts` before writing any claims.** Every concept in the title or centrally defined in the abstract must correspond to a DEFINITION in `claims`.
2. **Method** — `default_method` and `default_causal_strength` for the paper. **Judge causal_strength by research design, not the author's word choice.**
3. **Results + Tables + Figures** — each Table maps to at least one RELATION or EXISTENCE_CLAIM. Read numbers directly from Tables. Record **all baselines and comparison models**, not just the winner. **For every hypothesis (H1, H2, ...), produce a RELATION claim with `verification_status` reflecting what the Results say about that hypothesis.**
4. **Discussion** — author interpretation, not data. Usually `verification_status = proposed_in_paper`. Override `causal_strength` only when different from paper defaults.
5. **References** — 10-25 substantively used. Do not fabricate DOIs.

## Hypothesis-by-hypothesis coverage

**Papers with numbered hypotheses (H1, H2, ..., Hn) must produce one RELATION claim per hypothesis.** The `verification_status` of each hypothesis-RELATION reflects its outcome in Results:

| paper outcome | verification_status | note |
|---------------|---------------------|------|
| H supported by data | `verified_in_paper` | standard |
| H supported for one DV, not another | `mixed_support` (or split into two claims) | capture the split |
| H tested but NOT supported | `not_supported` | **do not skip these** |
| H proposed but not tested | `proposed_in_paper` | theoretical only |
| H supported only partially (effect sign wrong, or ns) | `partial_support` | note the specifics |

When the paper's summary section lists "H1-H8 were supported except H5 and H8," you must produce 8 RELATION claims, with `verification_status` corresponding to each. Missing an H# here is a silent failure — downstream users will think the paper didn't address that hypothesis at all.

## Output: single JSON object

```json
{"paper": {...}, "claims": [...], "theories": [...], "measurement_methods": [...], "claim_relations": [...], "paper_references": [...]}
```

### 1. paper

```json
{
  "doi": "10.xxxx/yyyy",
  "title": "...",
  "title_concepts": ["{unit}_{concept}", "..."],
  "authors": ["..."],
  "year": 2024,
  "venue": "...",
  "paper_type": "empirical | conceptual | review | meta_analysis | book | book_chapter",
  "domain": "e.g. organizational_behavior, ml, ai_safety",
  "default_method": {
    "sample_size": N,
    "unit_of_analysis": "team | individual | organization | computational | gene | national | dyadic | ...",
    "industry": "...",
    "country": "...",
    "study_design": "..."
  },
  "default_causal_strength": {
    "design_basis": "cross_sectional | longitudinal | quasi_experimental | rct | meta_analysis | computational_demonstration | theoretical",
    "temporal_precedence": "none | partial | full",
    "author_framing": "causal | associational | suggestive | existence_proof",
    "manipulation_of_iv": true/false
  },
  "hypotheses_tested": ["H1", "H2", "H3", "..."]
}
```

**`title_concepts` is written before claims** so each concept in `title_concepts` has an accountable DEFINITION.

**`hypotheses_tested` is a list of hypothesis numbers explicitly proposed and tested in the paper.** This is a checklist: for each item listed, there must be a corresponding RELATION claim later. Write this BEFORE generating claims, so the claims array is constrained by coverage accountability.

### 2. claims — 4 templates

Every claim:

```json
{
  "id": "{first_author_surname_lc}{year}_{def|rel|exist|meta}{n}",
  "template_type": "DEFINITION | RELATION | EXISTENCE_CLAIM | META_CLAIM",
  "l0_json": {...},
  "l3_json": {...},
  "source_text": "1-3 sentence direct quote (no paraphrasing)",
  "source_page": <printed page number>,
  "source_section": "...",
  "evidence_type": "see Appendix A",
  "verification_status": "verified_in_paper | mixed_support | partial_support | not_supported | proposed_in_paper | null",
  "confidence_score": 0.0-1.0,
  "causal_strength": {...},
  "method_metadata": {...}
}
```

**No `l1` or `l2` fields.** Clients generate natural language and triples from `l0_json + source_text` at read time.

**Override rules**: `causal_strength` and `method_metadata` are included in a claim **only when they differ from paper defaults**. Omit entirely when paper default applies.

Minimum counts: empirical ≥ 5, conceptual ≥ 8. If a paper numbers N hypotheses, the number of hypothesis-RELATIONs must equal N.

#### DEFINITION — "what X is"

```json
"l0_json": {
  "concept": "{unit}_{concept}",
  "key_elements": ["..."],
  "distinguished_from": ["..."]
}
```

`verification_status = null`. For framework papers proposing multiple components: **create a separate DEFINITION for each component**, not one DEFINITION describing the whole framework.

#### RELATION — "X is related to Y"

```json
"l0_json": {
  "iv": "{unit}_{concept}",
  "dv": "{unit}_{concept}",
  "relation": "positive | negative | null | curvilinear | conditional | not_supported",
  "mediator": null,
  "moderator": null,
  "hypothesis_label": "H1",
  "author_generalization": null
}
```

`iv` and `dv` names are based on **what was actually measured in Method**, not the abstract label in Discussion. When Discussion over-generalizes:

```json
"author_generalization": {
  "discussion_label": "wellbeing",
  "actual_measure": "individual_life_satisfaction_swls"
}
```

`hypothesis_label` ties the RELATION to the paper's own numbering when the paper explicitly tests H1-Hn. Use `null` for relations not tied to a numbered hypothesis.

**`relation: "not_supported"`** captures the case where the paper proposed a direction (e.g., H8: "efficacy mediates X→Y") but Results rejected it. Pair with `verification_status: "not_supported"`.

#### EXISTENCE_CLAIM — "X exists / was observed / has a limitation"

```json
"l0_json": {
  "phenomenon": "...",
  "evidence": "1-2 sentence evidence"
}
```

**EXISTENCE_CLAIMs are the most underextracted template.** They preserve:

- Study design limitations (cross-sectional admissions, sample-size caveats) → without these, downstream users over-interpret RELATION claims
- Paper motivation / problem statement
- Capability demonstrations ("Method achieves SOTA")
- Boundary conditions
- Non-relational numerical findings (ICC, reliability, between-group variance, response rates)

Produce EXISTENCE_CLAIMs for **sample sizes and response rates** when reported as distinct facts (e.g., "N=51 teams, 427 individuals, 86% response rate").

#### META_CLAIM — "integrative finding across studies or across the paper"

```json
"l0_json": {
  "integrated_finding": "...",
  "synthesis_type": "quantitative_meta_analysis | qualitative_synthesis | theoretical_integration",
  "k": null,
  "N_total": null
}
```

### 3. theories

Record only **explicitly named** theories:

```json
[{
  "name": "...",
  "type": "theory | theoretical_model | framework",
  "component_claims": ["claim_id_1", "claim_id_2"],
  "description": "1-2 sentences",
  "scope": "..."
}]
```

### 4. measurement_methods

```json
[{
  "id": "short_name",
  "name": "...",
  "subtype": "questionnaire_scale | behavioral_observation | computational_benchmark | physiological_measure | experimental_paradigm | interview | content_analysis | archival_data",
  "measures": "concept_name",
  "details": {
    "item_count": N,
    "response_scale": "...",
    "reverse_items": [1, 3, 5],
    "reliability_reported": 0.82,
    "original_source": "...",
    "copyright_status": "public | proprietary"
  },
  "used_in_claims": ["..."]
}]
```

`reverse_items`: 1-based item numbers, not item text. `reliability_reported`: numeric value (e.g., `0.82`), not a string.

### 5. claim_relations

**claim_relations are what make this a knowledge graph.** Without them, downstream users cannot trace how one finding builds on, qualifies, or contradicts another — the reasoning structure is lost.

Record every relationship you find. Use `confidence_score` for uncertainty rather than omitting.

```json
[{
  "source_id": "...",
  "target_id": "...",
  "relation_type": "supports | extends | qualifies | contradicts | replicates | aggregates | part_of_model | background | apparent_tension",
  "incompleteness_category": "effect_disappears_under_control | hub_component | boundary_condition | extends_prior_definition | none",
  "confidence_score": 0.85
}]
```

**Framework papers**: connect each component → whole with `part_of_model` + `hub_component`.

**Motivation → proposal**: `background`.

**Mediation findings where direct effect vanishes**: `qualifies` + `incompleteness_category: effect_disappears_under_control`.

**Contradictory hypotheses in the same paper**: e.g., H7 (PS mediates) supported, H8 (efficacy mediates) not supported — link H7 ↔ H8 with `apparent_tension` if relevant.

### 6. paper_references

Only substantively used references (10-25 typical). Do not fabricate DOIs. Use `null` if uncertain.

```json
[{"cited_doi": "10.xxxx/yyyy", "cited_title": "..."}]
```

## Judgment

### Template selection

| pattern | template |
|---------|----------|
| "What X is" | DEFINITION |
| "X is related to Y" (with or without statistics) | RELATION |
| "X was tested — data shows it holds / fails" | RELATION with `verification_status` |
| "X exists / was observed / has a limitation / achieves Y" | EXISTENCE_CLAIM |
| "Integrative finding across studies or the paper" | META_CLAIM |

### causal_strength (paper-level default)

Judge by **research design**, not word choice:

- Benchmark / ablation → `temporal_precedence=none`, `author_framing=existence_proof`
- "outperforms", "SOTA" → `author_framing=existence_proof`
- "may", "could", "will be" → `author_framing=suggestive`
- Cross-sectional data, author writes "causes" → `design_basis=cross_sectional, author_framing=causal` (record the gap)

### verification_status

- Backed by Results/data → `verified_in_paper`
- Mixed (supported on one DV, not another) → `mixed_support`
- Partially supported (direction correct but ns) → `partial_support`
- Tested and rejected by data → `not_supported`
- Author's interpretation or theoretical extension → `proposed_in_paper`
- DEFINITION → `null`

### iv/dv naming

- `{unit_of_analysis}_{concept}` format
- Based on **Method measurements**, not Discussion abstractions
- Measurement-method differences (self-report vs observer) → `measurement_methods`, NOT `iv/dv`

## Appendix A: `evidence_type`

```
meta_analysis, rct, quasi_experimental, longitudinal_field,
cross_sectional_field, qualitative_field, computational_demonstration,
theoretical_derivation, theoretical_integration, re_interpretation,
expert_framework, conceptual_argument
```

## Appendix B: `l3_json` fields

**Statistical values**: `effect_size`, `effect_size_type`, `p`, `ci_lower`, `ci_upper`, `n`, `r_squared`, `r_squared_type`, `k`, `N_total`, `heterogeneity_I2`, `publication_bias`

**Structured data**:
- `mediation`: `{total_effect, direct_effect, indirect_effect, direct_p, indirect_p, proportion_mediated, note}`
- `interaction`: `{variable: value}`
- `models`: `[{label, effect_size, effect_size_type, ...}]` ← **all baselines**
- `reliability`: `{alpha, icc, ...}`
- `formal`: `{equation, variables, conditions, proof_type}`
- `additional`: `{any structural information}` ← required for conceptual papers

**`effect_size_type`**:
- Psychology: `B, beta, r, rho, d, eta_squared, partial_eta_squared, omega_squared, odds_ratio, hazard_ratio`
- ML: `BLEU, accuracy, F1, perplexity, Exact_Match, ROUGE, AUC, MAE, MSE`

## Appendix C: `incompleteness_category`

```
effect_disappears_under_control — effect vanishes when controlling
hub_component                   — component of mediation/moderation model
boundary_condition              — a boundary condition has been stated
extends_prior_definition        — extends a prior definition
none                            — standalone citation safe
```

## Appendix D: concept naming

`{unit_of_analysis}_{concept}`. Prefixes: `team_`, `individual_`, `organizational_`, `computational_`, `architecture_`, `channel_`, `source_`, `cognitive_`, `agent_`, `safety_`, `therapeutic_`, `gene_`, `national_`, `dyadic_`.

**Hints (prefer these names when the concept matches)**:

- OB team-level: `team_psychological_safety`, `team_learning_behavior`, `team_performance`, `team_context_support`, `team_leader_coaching`, `team_efficacy`, `team_type`
