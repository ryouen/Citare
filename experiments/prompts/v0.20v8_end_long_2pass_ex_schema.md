# Citare v0.20-V8 (L8 cell: end / long / two-pass / example / schema-mandatory discipline)

## Purpose of This Task

You will read an academic paper and extract its "claims" as structured JSON for the Citare Graph database.

**Why this matters:**
The same paper is read millions of times by researchers and AI agents worldwide. Once structured, it never needs to be re-processed. The more accurate and comprehensive your extraction, the more every subsequent user benefits.

**The two errors we most want to prevent:**
1. **Causal misattribution** — writing "causes" when the paper only shows "is associated with." The causal_strength field prevents this.
2. **Missing important claims** — a key finding absent from the structured data. Missing claims are extremely hard to detect after the fact. When in doubt, output with a lower confidence_score rather than omitting.

Read all text content, tables, and equations from the PDF, including what is rendered only as images when necessary.

---

## Output Structure

Output everything as a single valid JSON object.

### 1. paper (metadata + study-level defaults)

```json
{
  "paper": {
    "doi": "10.xxxx/yyyy",
    "title": "Paper title",
    "title_concepts": ["concept_a", "concept_b"],
    "authors": ["Author A", "Author B"],
    "year": 2024,
    "venue": "Journal name",
    "paper_type": "empirical / conceptual / review / meta_analysis / book / book_chapter",
    "domain": "e.g. organizational_behavior, machine_learning, ai_safety",

    "default_method": {
      "sample_size": 51,
      "unit_of_analysis": "e.g. team, individual, organization, computational, gene, national, dyadic",
      "industry": "e.g. manufacturing, healthcare, mixed",
      "country": "e.g. US, JP, multinational",
      "study_design": "e.g. cross-sectional survey, RCT, benchmark evaluation"
    },

    "default_causal_strength": {
      "design_basis": "cross_sectional / longitudinal / quasi_experimental / rct / meta_analysis / computational_demonstration / theoretical",
      "temporal_precedence": "none / partial / full",
      "author_framing": "causal / associational / suggestive / existence_proof",
      "manipulation_of_iv": false
    }
  }
}
```

**default_method** and **default_causal_strength** describe the primary study design. Most claims in a single paper share the same research design and sample. Individual claims only need to override these when they differ (e.g., a different unit_of_analysis for one analysis, or a Discussion-only claim with design_basis = "theoretical").

**title_concepts:** List concepts from the title or abstract that the paper substantively defines, using `{unit_of_analysis}_{concept}` format. Each listed concept must have a corresponding DEFINITION claim in the claims array. Do not list generic English words (e.g. "intelligence" in a philosophy paper) unless the paper defines them specifically.

**DOI:** Search the PDF's headers, footers, and body text. If not printed but confidently inferrable, record with `"doi_source": "inferred"`. Use JSON `null` only when truly unidentifiable.

**source_page (all claims):** Use the **printed page number** from headers/footers, not the PDF viewer's page count. If absent, use PDF page numbers and add `"source_page_note": "PDF page number"` to the claim.

### 2. claims — four templates

The atomic unit of Citare. For short papers, at least one claim per substantive paragraph or Table/Figure. For longer papers, one claim per section or major finding is sufficient.

Minimum counts: empirical ≥ 5, conceptual ≥ 8. A paper with fewer substantive sections may produce fewer claims.

#### Common fields

```json
{
  "id": "{first_author_surname_lowercase}{year}_{def|rel|exist|meta}{number}",
  "template_type": "DEFINITION | RELATION | EXISTENCE_CLAIM | META_CLAIM",
  "l0_json": {},
  "l3_json": {},
  "source_text": "Direct quote of 1-3 sentences from the paper. No paraphrasing, no added commentary.",
  "source_page": 354,
  "source_section": "Section name",
  "evidence_type": "See Appendix A",
  "verification_status": "verified_in_paper | proposed_in_paper | null (JSON null, for DEFINITION)",
  "confidence_score": 0.90
}
```

**No l2 or l1 fields.** Citare clients generate natural-language summaries and triples from l0_json + source_text at read time.

#### DEFINITION — "What X is"

```json
"l0_json": {
  "concept": "{unit_of_analysis}_{concept}",
  "key_elements": ["element1", "element2", "element3"],
  "distinguished_from": ["concept it could be confused with"]
}
```

`verification_status` is JSON null for DEFINITION.

For framework papers: create **separate DEFINITIONs for each component**, not just one for the whole framework.

#### RELATION — "X is related to Y"

```json
"l0_json": {
  "iv": "{unit_of_analysis}_{concept}",
  "dv": "{unit_of_analysis}_{concept}",
  "relation": "positive | negative | null_effect | curvilinear | conditional",
  "mediator": null,
  "moderator": null,
  "author_generalization": null
}
```

**`relation` values:**
- `positive` / `negative`: monotonic direction
- `null_effect`: the paper tested the relation and found no significant effect (the relation is claimed to be absent)
- `curvilinear`: U-shaped / inverted-U / other non-monotonic
- `conditional`: effect depends on context; prefer setting `moderator` instead if you know the moderator

Use JSON `null` (not the string "null_effect") for `mediator` / `moderator` / `author_generalization` when those are not applicable.

**author_generalization:** Use JSON `null` when Discussion and Method use the same terms. When the author uses a more abstract label in Discussion than what was measured in Method:

```json
"author_generalization": {
  "discussion_label": "wellbeing",
  "actual_measure": "individual_life_satisfaction_diener_swls"
}
```

**causal_strength override:** Include this field in a claim **only** when it differs from paper.default_causal_strength. Only include the keys that differ. Omit the entire field when the paper default applies.

```json
"causal_strength": {
  "design_basis": "theoretical",
  "author_framing": "suggestive"
}
```

**method_metadata override:** Include **only** when different from paper.default_method. Only include the keys that differ. Omit the entire field when the paper default applies.

```json
"method_metadata": {
  "sample_size": 427,
  "unit_of_analysis": "individual"
}
```

**l3_json:** Record only applicable fields (full list in Appendix B). Always reference the paper's Tables directly and record **all comparison baselines**. Record equations in `formal`. For conceptual papers, record structural information in `additional`.

#### EXISTENCE_CLAIM — "X exists / was observed / has a limitation"

EXISTENCE_CLAIMs capture findings that are not variable relationships but are critical for correct interpretation — without them, downstream users may over-interpret RELATION claims. Study design limitations, boundary conditions, and methodological validations belong here.

```json
"l0_json": {
  "phenomenon": "phenomenon name",
  "evidence": "1-2 sentence summary of evidence"
}
```

**Commonly overlooked but important cases.** Each of these, when present in a paper, should produce its own EXISTENCE_CLAIM — do not fold them silently into the Method section:

- **Study design limitations explicitly acknowledged by the author.** Example: "The cross-sectional design prevented a demonstration of causality." This is the single most important claim for downstream citation safety, because it bounds what RELATION claims from the same paper can be legitimately interpreted as.
- **Paper motivation / problem statement** ("Current Web lacks machine-processable semantics"). One EXISTENCE_CLAIM per distinct motivation paragraph in the Introduction.
- **Capability demonstration** ("Non-parametric memory can be hot-swapped").
- **Boundary condition** ("The method failed on personality tests").
- **Aggregation justifications** — if the paper reports ICC or between-group variance to justify team-level / group-level analysis, that is an EXISTENCE_CLAIM (not just a methodological footnote). Example: `phenomenon: between_team_variance_justifies_aggregation`, `evidence: "ICC=.39, F(50,427)=6.98, p<.001"`.
- **Sample sizes and response rates** when reported as distinct facts. Example: "51 teams, 427 individuals, 86% response rate" — produce an EXISTENCE_CLAIM even when the numbers also appear in `paper.default_method`.
- **Non-relational numerical findings** (reliability alphas reported outside a scale table, between-group variance, null-finding ICCs).

#### META_CLAIM — "Integrative finding across studies or across the entire paper"

```json
"l0_json": {
  "integrated_finding": "Summary of integrated finding",
  "synthesis_type": "quantitative_meta_analysis | qualitative_synthesis | theoretical_integration",
  "k": null,
  "N_total": null
}
```

### 3. theories

Record only theories **explicitly named** in the paper. Criteria: (1) organizes 3+ claims, (2) includes mechanism, (3) can generate hypotheses.

```json
"theories": [{
  "name": "Formal name",
  "type": "theory | theoretical_model | framework",
  "component_claims": ["claim ID list"],
  "description": "1-2 sentences",
  "scope": "Applicable range"
}]
```

### 4. measurement_methods

```json
"measurement_methods": [{
  "id": "short_name",
  "name": "Full name",
  "subtype": "questionnaire_scale | behavioral_observation | computational_benchmark | physiological_measure | experimental_paradigm | interview | content_analysis | archival_data",
  "measures": "concept name",
  "details": {
    "item_count": null,
    "response_scale": null,
    "reverse_items": [1, 3, 5],
    "reliability": {"type": "cronbach_alpha | icc | kappa | test_retest | other", "value": 0.82},
    "original_source": "Author, year",
    "copyright_status": "public | proprietary"
  },
  "used_in_claims": ["claim ID list"]
}]
```

- `reverse_items`: list of 1-based item numbers, not item text (e.g., `[1, 3, 5]` not `["If you make a mistake..."]`).
- `reliability.value`: numeric value (e.g., `0.82`), not a string. `reliability.type` disambiguates which metric (Cronbach's alpha vs intraclass correlation vs kappa).

### 5. claim_relations

claim_relations are what make Citare a graph, not just a list of facts. Without them, users cannot trace how one finding builds on, qualifies, or contradicts another — the reasoning structure of the paper is lost. Record all relationships you find. Use `confidence_score` to express uncertainty rather than omitting.

```json
"claim_relations": [{
  "source_id": "claim ID",
  "target_id": "claim ID",
  "relation_type": "supports | extends | qualifies | contradicts | replicates | aggregates | part_of_model | background | apparent_tension",
  "incompleteness_category": "effect_disappears_under_control | hub_component | boundary_condition | extends_prior_definition | none",
  "confidence_score": 0.85
}]
```

The meaning of each relation is fully expressed by relation_type + incompleteness_category + the content of the source and target claims.

Framework papers: connect components → whole with `part_of_model` + `hub_component`.
Motivation → proposal: connect with `background`.

### 6. paper_references

Record references that are substantively used in the paper, not bulk citation lists.

```json
"paper_references": [
  {"cited_doi": "10.xxxx/yyyy", "cited_title": "Title"},
  {"cited_doi": null, "cited_title": "Title only"}
]
```

**Record** if findings, definitions, or frameworks from the reference are used to build a claim; if results are compared or extended; or if cited with specific findings.

**Do not record** parenthetical lists without individual discussion. Typical count: 10-25.

DOIs: record only when certain. Use JSON `null` otherwise; do not fabricate.

## How to Read the Paper

Read in this order. Each step produces specific output.

**Step 1 — Title + Abstract.** Extract paper metadata including title_concepts. Paper motivations become EXISTENCE_CLAIMs.

**Step 2 — Method.** Extract default_method and default_causal_strength for the paper. Judge causal_strength by research design, not the author's word choice. Identify measurement_methods.

**Step 3 — Results + Tables + Figures.** Each Table maps to at least one RELATION or EXISTENCE_CLAIM. Read numbers directly from Tables. For each framework, create separate DEFINITIONs for each component, not one for the whole.

**Step 4 — Discussion.** Author interpretation, not data. Usually verification_status = "proposed_in_paper". Override causal_strength or method_metadata only when different from paper defaults.

**Step 5 — References.** Select 10-25 substantively used references. Record cited_doi only when certain.

## Judgment Criteria

**Template selection:**
```
"What X is"                                          → DEFINITION
"X is related to Y" (with or without statistics)     → RELATION
"X exists / was observed / has a limitation"         → EXISTENCE_CLAIM
"Integrative finding across studies or the paper"    → META_CLAIM
Citing another field's law as evidence for argument  → Independent RELATION with evidence_type: re_interpretation
```

Example of `re_interpretation`: a conceptual paper applying the Second Law of Thermodynamics as an analogy for information decay in a social system. The law itself is a RELATION; its re-use here is an independent RELATION with evidence_type = re_interpretation.

**causal_strength judgment (applied at paper.default_causal_strength level):**
- Judge by **research design**, not the author's word choice
- Benchmark comparisons / ablations → temporal_precedence = "none", author_framing = "existence_proof"
- "outperforms", "achieves SOTA" → author_framing = "existence_proof"
- "will be realized", "may influence" → author_framing = "suggestive"
- Cross-sectional data with causal wording in Discussion → design_basis = "cross_sectional", author_framing = "causal" (record the gap truthfully)

**verification_status:**
- Backed by Results/data → `verified_in_paper`
- Author's interpretation / untested explanation → `proposed_in_paper`
- Fact and interpretation mixed in the same passage → split into two claims
- DEFINITION → JSON `null`

**iv/dv naming:**
- Use `{unit_of_analysis}_{concept}` format, where the prefix matches the study's unit of analysis (e.g. `team_`, `individual_`, `gene_`, `national_`)
- Use the paper's own terminology faithfully for the concept part
- If Appendix D lists a matching concept, prefer that name for consistency
- Based on **Method section measurements**, not Discussion abstractions
- Gap between Method and Discussion → record in `author_generalization`
- Measurement differences (self-report vs observer-rated) → measurement_methods, not iv/dv

## Appendix A: evidence_type (12 types)

```
meta_analysis              — aggregation across studies with statistics
rct                        — randomized controlled trial
quasi_experimental         — experimental manipulation without full randomization
longitudinal_field         — field data over multiple time points
cross_sectional_field      — field data at a single time point
qualitative_field          — interviews, observation, case study
computational_demonstration— benchmark, simulation, algorithmic evaluation
theoretical_derivation     — derivation from axioms or first principles
theoretical_integration    — synthesis of prior theories into a new framework
re_interpretation          — applying another field's result to this context
expert_framework           — framework proposed from professional experience
conceptual_argument        — conceptual reasoning without formal derivation
```

## Appendix B: l3_json Available Fields

Record only applicable fields. Do not fill with JSON `null`s.

**Statistical values:**
```
effect_size, effect_size_type, p, ci_lower, ci_upper, n,
r_squared, r_squared_type, k, N_total, heterogeneity_I2, publication_bias
```

**Structured data:**
```
mediation: {total_effect, direct_effect, indirect_effect, direct_p, indirect_p, proportion_mediated, note}
interaction: {variable: value}
models: [{label, effect_size, effect_size_type, ...}]  ← record ALL baselines
reliability: {alpha, icc, ...}
formal: {equations: [{latex, variables: {symbol: desc}, conditions, proof_type, name, equation_status}]}
additional: {any structural information}  ← required for conceptual papers
```

**effect_size_type standard values:**

Psychology: `B, beta, r, rho, d, eta_squared, partial_eta_squared, omega_squared, odds_ratio, hazard_ratio`
ML: `BLEU, accuracy, F1, perplexity, Exact_Match, ROUGE, AUC, MAE, MSE`

## Equations — two-pass protocol + mandatory `equation_status` field

**Pass 1 (primary)**: extract every claim (DEFINITION/RELATION/EXISTENCE_CLAIM/META_CLAIM) from the paper without attaching any equation.

**Pass 2 (equation attachment)**: revisit each claim. For any claim whose source_text directly states or embeds an equation, attach that equation to `formal.equations`. Do NOT create new claims to host equations. Do NOT extract equations that are not already covered by an extracted claim.

Example: `{"latex": "n^{*} = (\\eta_{\\mathrm{tail}}/\\eta_c)^{-1/\\tau}", "equation_status": "central_contribution", "name": "crossover", "proof_type": "direct_derivation"}`



Every equation object MUST include an `equation_status` field with EXACTLY one of:

- `"central_contribution"` — the equation IS the paper's own theorem / claim / result / proposed method.
- `"supporting_definition"` — the equation defines a construct this paper introduces (used by a `central_contribution` equation).
- `"restatement"` — simplified or paraphrased form of an equation stated elsewhere in the same paper.
- `"textbook_background"` — canonical form cited from prior work (standard scaling law, Rademacher, Hoeffding, standard definitions).

**Output policy**: include ONLY equations with `equation_status` in `{central_contribution, supporting_definition}`. Do NOT include equations classified as `restatement` or `textbook_background` — these are present in the paper but are not the paper's own contribution. If you cannot confidently classify an equation as `central_contribution` or `supporting_definition`, omit it.

LaTeX rules: no `$...$` wrappers; no ASCII-math (`\tau` not `τ`, `\sqrt{}` not `sqrt()`, `\sum_{i=1}^{n}`, `\frac{a}{b}`). One equation per list entry; `name` canonical if present; `proof_type` in `{definitional, direct_derivation, empirical_fit, axiomatic, postulated, assumed}`.

Equations are an ADDITIONAL signal. All claim templates (DEFINITION, RELATION, EXISTENCE_CLAIM, META_CLAIM) must be fully populated independent of equation presence.

Usage: `effect_size` = one primary effect measure per claim. `r_squared` = overall model fit. `reliability` = scale reliability. `additional` = anything else.

## Appendix C: incompleteness_category (5 types)

```
effect_disappears_under_control — effect vanishes when controlling for another variable
hub_component                   — component of a mediation/moderation model; citing alone loses the hub
boundary_condition              — a boundary condition has been stated; generality is limited
extends_prior_definition        — extends a prior concept; original must be noted
none                            — can be cited standalone without missing context
```
